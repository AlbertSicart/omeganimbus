02 · AWS CAF

AWS Cloud Adoption Framework

Definición
Enfoque integral para la adopción de la nube. Identifica funcionalidades organizativas concretas para realizar transformaciones correctas. Agrupa estas funcionalidades en 6 perspectivas.

Las 6 perspectivas se dividen en dos grupos: negocio (empresa, personas, gobernanza) y técnicas (plataforma, seguridad, operaciones).

03 · Adopción cloud

Etapas de adopción de la nube

1
Proyecto
Evaluar si AWS cubre tus requisitos. Fase exploratoria.
2
Fundamentos
Primera migración real. Apps de producción o landing zone en entorno no-prod.
3
Migración
Definir roles cloud ops, establecer CCoE, preparar operaciones a largo plazo.
4
Reinvención
Todos los proyectos nuevos comienzan en AWS. Transformación completa.

El CCoE (Cloud Center of Excellence) es el equipo interno responsable de las operaciones cloud a largo plazo.

04 · Estrategias de migración

Las 7 Rs de migración

Estrategia También conocida como Cuándo usarla Cambios en la app
Retirar — Aplicaciones que ya no se necesitan. Se eliminan. N/A
Retener — Apps que se quedan en origen o no están listas para migrar. Ninguno
Volver a alojarRehost "Lift and shift" Migrar sin modificar la aplicación. Rápido y sencillo. Ninguno
ReubicarRelocate — Gran número de servidores con una o varias apps. Mover en bloque. Mínimos
Volver a comprarRepurchase "Drop and shop" / Readquisición Cambiar a un producto diferente (SaaS) que aporta más valor que la infra actual. Nuevos procesos
Redefinir plataformaReplatform "Lift, tinker and shift" Apps que necesitan cierto nivel de optimización para aprovechar AWS. Moderados
RefactorizarRefactor / Re-architect Rediseñar Sacar el máximo partido de funciones cloud-native: agilidad, rendimiento, escalabilidad. Profundos

Herramientas de migración relevantes: AWS Database Migration Service (DMS) para replicación de BD · AWS Snowball para mover grandes volúmenes de datos.

05 · Servicios de BD para el examen

Cheatsheet de bases de datos AWS

Amazon RDS

  • Tipo: Relacional administrado
  • Motores: MySQL, PostgreSQL, Oracle, SQL Server, MariaDB
  • Cuándo: BD relacional con HA, backups y escalado gestionados
  • Keyword: "relacional"

Amazon DynamoDB

  • Tipo: NoSQL (clave-valor + documentos JSON)
  • Cuándo: Escalabilidad masiva, latencia predecible, sin esquema fijo
  • Keyword: "NoSQL", "no relacional"

Amazon Aurora

  • Tipo: Relacional administrado (AWS-native)
  • Motores: Solo MySQL y PostgreSQL
  • Cuándo: Alto rendimiento, compatible con MySQL/PG, coste optimizado vs RDS
  • ⚠ No soporta SQL Server

EC2 + AMI con SQL Server

  • Tipo: Self-managed
  • Cuándo: SQL Server Standard con coste optimizado (sin licencia propia mediante AMI de AWS)
  • Más barato que RDS para SQL Server
Almacenamiento para migración
Amazon S3 Glacier → archivado de grandes volúmenes a bajo coste. Tener en cuenta tarifas y tiempos de recuperación.
Amazon EFS → almacenamiento de archivos elástico y escalable para instancias EC2.
AWS Snowball Edge → dispositivo físico para mover grandes cantidades de datos hacia/desde AWS.
▲ Exam Tips — Puntos clave
  • CAF ≠ WAF: CAF = transformación organizativa (antes/durante migración). WAF = arquitectura técnica (una vez en AWS).
  • Las 6 perspectivas del CAF: Empresa, Personas, Gobernanza (negocio) + Plataforma, Seguridad, Operaciones (técnica).
  • Conocer las 7 Rs de memoria: Retirar, Retener, Rehost, Reubicar, Repurchase, Replatform, Refactor.
  • Lift and shift = Rehost. Sin cambios en la aplicación. La opción más rápida.
  • Refactor = mayor esfuerzo, mayor aprovechamiento de cloud-native. Opción más cara/compleja.
  • NoSQL / no relacional → DynamoDB. Relacional → RDS o Aurora.
  • Aurora no soporta SQL Server. Para SQL Server con coste optimizado → EC2 con AMI de SQL Server Standard.
  • Snowball = migración física de grandes volúmenes de datos. No es una BD.
  • S3 Glacier = almacenamiento de archivo de menor coste. Penalización en tiempo/coste de recuperación.
