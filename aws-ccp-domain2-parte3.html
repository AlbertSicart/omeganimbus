01 · Cuenta AWS y usuario raíz

Usuario raíz — Acceso total, uso mínimo

Usuario raíz (Root User)
Identidad creada al abrir una cuenta AWS. Tiene acceso completo y sin restricciones a todos los servicios y recursos. No se recomienda para tareas diarias. En su lugar, crear un usuario administrador en IAM Identity Center.

⚠ Cuándo SÍ usar root

  • Cambiar la configuración de la cuenta AWS
  • Restaurar permisos de usuario IAM
  • Activar acceso IAM a consola de facturación
  • Cerrar la cuenta AWS
  • Cambiar el plan de soporte de AWS

✓ Cómo proteger root

  • Activar MFA (autenticación multifactor)
  • Guardar credenciales de forma segura
  • Rotar claves de acceso y contraseña regularmente
  • No usarlo para tareas diarias
  • No crear claves de acceso para root
02 · Entidades IAM

Usuarios · Grupos · Roles · Políticas

👤 IAM User
Identidad permanente para una persona o aplicación con credenciales propias.
  • Nombre de usuario + contraseña (consola)
  • Access Key + Secret Key (acceso programático / CLI)
  • MFA habilitado por usuario
  • Políticas de complejidad y rotación de contraseñas
👥 IAM Group
Colección de usuarios IAM. Los permisos asignados al grupo los heredan todos sus miembros.
  • Facilita la gestión a escala
  • Un usuario puede pertenecer a varios grupos
  • Los grupos no pueden contener otros grupos
  • Ejemplo: grupo "Developers" con acceso a EC2 y S3
🎭 IAM Role
Credenciales temporales que pueden asumir usuarios, servicios AWS o entidades externas.
  • Un usuario asume un rol para acceso temporal
  • Un servicio AWS asume un rol para llamar a APIs
  • Acceso entre cuentas cruzadas (cross-account)
  • Identidades federadas / SSO
  • Preferible a usuarios IAM para servicios y apps
📋 IAM Policy
Documento JSON que define qué acciones están permitidas o denegadas sobre qué recursos.
  • Se adjunta a usuarios, grupos o roles
  • Efecto: Allow / Deny
  • Deny explícito siempre prevalece sobre Allow
  • Por defecto: todo denegado hasta que se permite
03 · Tipos de políticas IAM

Políticas administradas vs no administradas

Tipo Quién la crea Quién la gestiona Ejemplo
Administrada por AWS AWS AWS (actualizada automáticamente) AmazonS3ReadOnlyAccess
Administrada por cliente Cliente Cliente (control total) Política personalizada para un caso de uso específico
Inline Cliente Embebida directamente en usuario/grupo/rol Política de un solo uso vinculada a una entidad concreta
IAM Policy Simulator
Herramienta de AWS para probar y depurar políticas IAM y basadas en recursos antes de aplicarlas en producción. Permite simular qué acciones estarían permitidas o denegadas para un usuario, grupo o rol concreto.
04 · Decisión clave

¿Cuándo usar rol vs usuario IAM?

✓ Usar IAM Role cuando…

  • Un servicio AWS necesita acceder a otro (ej. EC2 → S3)
  • Acceso temporal para un usuario humano
  • Acceso entre cuentas AWS distintas
  • Federación de identidades (SSO, SAML, OIDC)
  • Aplicaciones o scripts que llaman a la API de AWS

✓ Usar IAM User cuando…

  • Persona individual con acceso permanente a la consola
  • Acceso programático de larga duración para una aplicación específica
  • Integración con sistemas externos que no soportan roles
05 · Identidades externas

Amazon Cognito — Usuarios externos y no autenticados

Cognito User Pool

Directorio de usuarios. Gestiona registro, login y MFA. Solo usuarios autenticados. No permite acceso a identidades no autenticadas.

Cognito Identity Pool

Proporciona credenciales AWS temporales a usuarios autenticados (redes sociales, SAML) y a usuarios invitados no autenticados.

Caso de uso típico: app móvil que permite acceso como invitado (sin login) y también con Google/Facebook → usar Identity Pool.

06 · Caso práctico — Amazon S3

Políticas de acceso en S3

Bucket Policy

Política basada en recurso adjunta al bucket. Permite o deniega acceso a usuarios IAM de cualquier cuenta AWS y a cuentas externas. Formato JSON.

User Policy (IAM)

Política adjunta al usuario IAM. Controla qué buckets y objetos puede usar ese usuario. Formato JSON idéntico.

Protección contra eliminación accidental en S3
Para evitar eliminación no autorizada de objetos:

1. Activar versionado de objetos en el bucket (requisito previo)
2. Activar MFA Delete → cualquier eliminación permanente requiere autenticación MFA adicional
▲ Exam Tips — Puntos clave
  • Root user = acceso total, usar solo para tareas específicas. Proteger con MFA obligatoriamente.
  • IAM User = identidad permanente. IAM Role = credenciales temporales. Preferir roles para servicios y apps.
  • Deny explícito siempre prevalece sobre cualquier Allow en las políticas IAM.
  • Por defecto, todo está denegado hasta que se concede explícitamente un permiso.
  • Políticas administradas por AWS → creadas y mantenidas por AWS. Políticas de cliente → creadas y mantenidas por ti.
  • IAM Policy Simulator = herramienta para probar y depurar políticas antes de aplicarlas.
  • Cognito User Pool = solo autenticados. Cognito Identity Pool = autenticados + invitados no autenticados.
  • MFA Delete en S3 requiere habilitar versionado primero.
  • Least privilege: conceder solo el acceso mínimo necesario. Aplicar siempre.
