01 · Amazon EC2

Arquitectura de EC2 — Componentes clave

Anatomía de una instancia EC2

Host EC2 (hardware físico)

Gestionado por AWS. Puede ser compartido (por defecto, aislado entre clientes) o dedicado (pagas por todo el host). La instancia vive en una AZ concreta.

Instance Store (almacén de instancia)

Almacenamiento local del host. Temporal: se pierde si la instancia se mueve a otro host, se detiene o falla.

Amazon EBS (almacenamiento persistente)

Volúmenes de bloque persistentes conectados a EC2 vía red de almacenamiento de la VPC. Resistencia zonal: solo conectables desde la misma AZ.

ENI — Elastic Network Interface

Interfaz de red asignada al host físico al lanzar la instancia. Se pueden añadir múltiples ENIs a una instancia para segmentación de red.

Instance Store vs EBS
Instance Store = temporal. Alta velocidad. Se pierde al detener/mover la instancia. Nunca usar para datos persistentes.
EBS = persistente. Sobrevive a reinicios. Vinculado a la AZ. No se puede conectar a instancias de otra AZ.
02 · Tipos de instancia EC2

5 categorías de instancias

Categoría Porción de recursos Casos de uso típicos Ejemplo familias
Uso general CPU y memoria equilibrados Cargas estacionarias, servidores web, entornos de desarrollo. Opción por defecto. t3, m6i
Computación optimizada Más CPU que memoria Procesamiento multimedia, machine learning, juegos, modelado científico, HPC. c6i, c7g
Memoria optimizada Más memoria que CPU Bases de datos en memoria, procesamiento de grandes datasets, SAP HANA. r6i, x2idn
Computación acelerada GPU / FPGA hardware Machine learning, rendering gráfico, deep learning, procesamiento de vídeo. p4, g5, f1
Almacenamiento optimizado Alta I/O disco Data warehousing, Elasticsearch, bases de datos analíticas, I/O secuencial y aleatorio. i4i, d3

Las instancias ampliables (tipo T) tienen créditos de CPU: carga normal baja + ráfaga temporal. Más económicas para cargas irregulares.

03 · AMI y datos de instancia

AMI · Metadatos · User Data

AMI (Amazon Machine Image)
Imagen que contiene SO, configuración y software preinstalado para lanzar instancias EC2. Una AMI dorada incluye parches de seguridad, agentes de monitorización, software corporativo y configuración estándar aprobada.

Instance Metadata

Datos sobre la instancia: ID de instancia, IP pública, clave pública, perfil de IAM, kernel…

Accesible desde dentro de la instancia vía:

http://169.254.169.254/latest/meta-data/

User Data

Scripts que se ejecutan al iniciar la instancia por primera vez. Para automatizar configuración, instalar paquetes, arrancar servicios.

No contiene información sobre la instancia (ID, IP, etc.).

http://169.254.169.254/latest/user-data/
04 · Opciones de computación más allá de EC2

Contenedores · Lambda · Elastic Beanstalk

📦 Amazon ECS
Orquestación de contenedores administrada por AWS. Acepta contenedores e instrucciones de dónde y cómo ejecutarlos.
Cuando quieres contenedores sin gestionar el plano de control de Kubernetes.
contenedores · orquestación · Docker
☸️ Amazon EKS
Kubernetes gestionado en AWS. Ejecuta clústeres Kubernetes sobre instancias EC2 o Fargate.
Cuando ya usas Kubernetes o necesitas su ecosistema completo.
Kubernetes · contenedores · clúster
λ AWS Lambda
FaaS (Function as a Service). Código ejecutado en respuesta a eventos. Sin gestión de servidores. Facturación por duración de ejecución (ms).
Lógica de negocio basada en eventos, automatizaciones, microservicios serverless.
serverless · event-driven · FaaS
🌱 AWS Elastic Beanstalk
PaaS que gestiona automáticamente la infraestructura (EC2, ELB, Auto Scaling). Solo subes el código.
Cuando quieres desplegar apps sin gestionar infraestructura pero manteniendo control sobre la configuración.
PaaS · deploy · código sin gestionar infra
EC2 vs Contenedores — ¿Por qué contenedores?
En EC2 cada instancia tiene su propio SO invitado completo → mucha memoria y disco consumida en duplicación de SO.
Los contenedores comparten el SO del host a través de un motor de contenedores. Un SO, múltiples procesos aislados → mucho más eficiente en recursos.
05 · Alta disponibilidad y elasticidad

Auto Scaling + Load Balancers

Usados conjuntamente para lograr alta disponibilidad, tolerancia a fallos y elasticidad. Auto Scaling añade/elimina instancias según demanda; el Load Balancer distribuye el tráfico entre ellas.

Tipo de ELB Capa OSI Protocolo Cuándo usarlo
Classic Load Balancer L4 / L7 HTTP, HTTPS, TCP Legado. Evitar para nuevas arquitecturas.
Application Load Balancer (ALB) L7 (aplicación) HTTP, HTTPS, WebSocket Apps web, microservicios, enrutamiento por URL/cabeceras/host. Opción más común para apps web.
Network Load Balancer (NLB) L4 (transporte) TCP, UDP, TLS Rendimiento extremo, baja latencia, IP estática, tráfico no-HTTP.
Gateway Load Balancer (GWLB) L3 (red) GENEVE Integrar appliances de red de terceros (firewalls, IDS/IPS) en el flujo de tráfico.
Nodo de ELB y alta disponibilidad
Al crear un ELB se crea un nodo de ELB por AZ. Para lograr alta disponibilidad real, debe haber un nodo en cada AZ que use la arquitectura. Los nodos se escalan horizontalmente de forma automática.
▲ Exam Tips — Puntos clave
  • EC2 = IaaS. Instancias son resistentes zonales (vinculadas a una AZ).
  • Instance Store = temporal (se pierde al detener). EBS = persistente, vinculado a la AZ.
  • EBS solo se puede conectar a instancias de la misma AZ.
  • 5 categorías de instancia: uso general · computación · memoria · acelerada · almacenamiento. Elegir según la carga.
  • Instance Metadata = info sobre la instancia (ID, IP, etc.) en 169.254.169.254/meta-data. User Data = scripts de arranque.
  • Lambda = FaaS, serverless, event-driven. Facturación por ms de ejecución.
  • ECS = orquestación de contenedores AWS-native. EKS = Kubernetes gestionado en AWS.
  • ALB = capa 7, apps web. NLB = capa 4, alto rendimiento/baja latencia. GWLB = appliances de red.
  • Auto Scaling + ELB juntos = elasticidad + alta disponibilidad + tolerancia a fallos.
