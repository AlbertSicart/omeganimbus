01 · Métodos de acceso y aprovisionamiento

Cómo interactuar con AWS

🖥️ Consola de administración AWS
Interfaz gráfica web. Ideal para exploración, configuración visual y tareas puntuales.
Mejor para: aprendizaje, tareas manuales, administración visual, troubleshooting rápido.
⌨️ AWS CLI
Interfaz de línea de comandos. Acceso a las APIs de AWS mediante comandos de texto desde terminal.
Mejor para: scripts, automatización, tareas repetitivas, pipelines CI/CD.
🧩 SDKs (programático)
Kits de desarrollo para integrar servicios AWS directamente en código (Python, Java, JS, Go…). Acceso a la API vía código.
Mejor para: aplicaciones que necesitan interactuar con AWS desde el código fuente.
📐 IaC — AWS CloudFormation
Infraestructura como código. Define y despliega recursos AWS mediante plantillas YAML/JSON declarativas.
Mejor para: entornos reproducibles, despliegues complejos, control de versiones de infraestructura.
02 · Modelos de despliegue — No confundir

Nube pública · Privada · Híbrida · Multi-cloud

☁ Nube pública

Plataforma cloud disponible para cualquier usuario que cumpla los criterios de computación en la nube. Gestionada por un proveedor externo.

AWS, Azure, Google Cloud

🏢 Nube privada

Servicios cloud dedicados exclusivamente a una organización, desplegados en sus propias instalaciones. Debe cumplir los 5 criterios cloud.

AWS Outposts, VMware Cloud

🔗 Nube híbrida

Combinación de nube pública + nube privada (ambas cumplen criterios cloud). No confundir con entorno híbrido.

AWS + AWS Outposts (on-prem con criterios cloud)

🌐 Multi-cloud

Uso simultáneo de más de una nube pública. Flexible pero complejo de diseñar y gestionar.

AWS + Azure, o AWS + GCP, o los tres
⚠ Distinción crítica para el examen
Nube híbrida = nube pública + nube privada (ambas cumplen criterios cloud).
Entorno híbrido = AWS + datacenter on-premises tradicional (el datacenter no es una nube privada).

Si las aplicaciones trabajan conjuntamente entre AWS y un datacenter local → es un entorno híbrido, no una nube híbrida.
03 · Arquitectura de red AWS

Servicios públicos vs privados en AWS

En AWS, "público" y "privado" hacen referencia a la red, no a quién tiene acceso. Es una distinción de conectividad, no de permisos.

Zonas de red en AWS
Internet pública
Tráfico abierto global
⟷
Zona pública AWS
Conectada a internet pública
Amazon S3, CloudFront, API Gateway
✕
Zona privada AWS
Sin conexión directa a internet
Amazon VPC, EC2 (por defecto), RDS
La zona privada puede conectarse a la zona pública o a internet si se configura explícitamente (Internet Gateway, NAT Gateway, etc.). Por defecto está completamente aislada.
Servicio público vs privado de AWS
Servicio público (ej. S3): accesible desde cualquier lugar con internet. Se encuentra en la zona pública de AWS.
Servicio privado (ej. EC2 en VPC): aislado por defecto. Sin acceso a internet ni a la zona pública salvo configuración explícita.

La zona pública de AWS no es internet: es una red propia de AWS adyacente a internet.
04 · Opciones de conectividad

Internet pública · VPN · Direct Connect · Gateways

Método Tipo Características Cuándo usarlo
Internet pública Pública Acceso estándar a servicios AWS públicos vía HTTP/S. Acceso general a servicios públicos como S3 o APIs.
AWS VPN Cifrada Túnel cifrado sobre internet pública entre red on-premises y VPC. Conectar datacenter local a AWS de forma segura y económica.
AWS Direct Connect Privada dedicada Conexión física dedicada y privada. No pasa por internet pública. Latencia predecible y consistente. Cargas de trabajo críticas con requisitos de ancho de banda alto, baja latencia o conformidad estricta.
Internet Gateway (IGW) VPC → Internet Puerta de enlace redundante y altamente disponible. Permite a instancias públicas en VPC comunicarse con internet. Conectar subredes públicas de una VPC a internet pública.
NAT Gateway Privado → Internet Permite a instancias en subredes privadas iniciar conexiones salientes a internet, sin exponer su IP. Instancias privadas que necesitan acceso a internet (actualizaciones, etc.) sin ser accesibles desde fuera.
Pregunta de examen frecuente
"¿Qué usas para conectar instancias EC2 en una subred pública a internet?"
→ Internet Gateway (IGW). Redundante, se escala horizontalmente, permite tráfico bidireccional entre VPC e internet.

IGW vs NAT Gateway: IGW es para subredes públicas (tráfico bidireccional). NAT Gateway es para subredes privadas (solo tráfico de salida iniciado desde dentro).
▲ Exam Tips — Puntos clave
  • Consola = visual/manual. CLI = scripts/automatización. SDK = desde código. CloudFormation = IaC declarativo.
  • Nube híbrida ≠ entorno híbrido. Híbrida = pública + privada (ambas cloud). Híbrido = AWS + datacenter tradicional.
  • AWS Outposts = nube privada on-premises de AWS. Cumple criterios cloud, dedicado a tu empresa.
  • Servicio público de AWS (S3) = accesible desde internet. Servicio privado (EC2 en VPC) = aislado por defecto.
  • Direct Connect = conexión privada dedicada. No pasa por internet. Baja latencia, predecible.
  • Internet Gateway = subred pública ↔ internet (bidireccional).
  • NAT Gateway = subred privada → internet (solo salida). Las instancias privadas no son accesibles desde fuera.
  • La zona pública de AWS no es internet: es la red interna de AWS con conectividad hacia internet.
